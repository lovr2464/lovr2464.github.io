关于ssm的理解

关于Spring体系，它无疑是庞大的，涵盖了从核心容器到各种子框架几乎所有Java开发中可能需要的功能。而每个Java程序员绕不开的基础便是SSM：Spring、Spring MVC、MyBatis。

此篇，我不谈代码，只谈理解，我也是在学中。

（本文大部分在讲spring mvc）

MVC与spring

MVC（Model-View-Controller，模型-视图-控制器）是软件工程中的一种软件架构模式，它把软件系统分为三个基本部分：模型（Model）、视图（View）、控制器（Controller）。大多数web开发都基于这种模式划分。

  • 视图层 (View)

    • 由前端实现：在MVC架构中，视图层通常由前端技术实现，负责将数据展示给用户。前端开发者使用HTML、CSS、JavaScript以及相关的前端框架（如React、Vue.js、Angular等）来构建用户界面。

  • 控制层 (Controller)

    Spring框架将控制层进一步分为表述层（Web层）和业务逻辑层。通过这种分层架构，开发者可以将业务逻辑与控制逻辑分开，使代码更加清晰、易于维护，并且提高了代码的可测试性。

    • **表述层 (Web层)**：

    ​ •职责：处理HTTP请求，调用业务逻辑层，选择适当的视图来响应请求。

    ​ •实现方式：通常由带有@Controller或@RestController注解的类实现，负责映射URL到具体的处理方法。表述层主要负责接收用户的请求，并将请求数据传递给业务逻辑层。

    • **业务逻辑层 (Service层)**：

    ​ •职责：执行应用程序的核心业务逻辑，处理来自表述层的数据，并与持久化层交互以获取或存储数据。

    ​ •实现方式：由带有@Service注解的类组成。业务逻辑层的类通常包含了主要的业务规则和流程，调用持久化层获取和保存数据。

  • 模型层 (Model)

    模型层可以进一步划分为POJO和数据库持久化（dao）部分：

    • **POJO (Plain Old Java Objects)**：

    ​ •职责：POJO代表了应用程序中的数据对象。这些类通常没有特殊的行为，只是简单地封装了数据。它们可能与数据库中的表结构对应。

    ​ •使用场景：POJO通常被业务逻辑层和持久化层使用，用于传递数据。

    • 数据库持久化部分：

    ​ •职责：负责与数据库的交互，包括CRUD操作（创建、读取、更新、删除）。

    ​ •实现方式：通常由DAO（Data Access Object）层实现，使用@Repository注解标识。

请求与响应的过程

我们先讲解一下原生的jakarta接收请求与响应的过程。简单了解一下即可。

  • 首先，用户通过自己的浏览器发出某种http请求，可能带参数也可能不带
  • 然后服务端服务器软件将接收到请求，同时服务器软件也是Jakarta应用运行的容器
  • 请求到达服务器并通过过滤器进行预处理，过滤器负责执行如身份验证、日志记录等操作。
  • 经过过滤器处理后，请求到达Servlet，servlet用于处理HTTP请求并生成响应。在此，你可以调用自己写的方法等处理请求，或者直接书写处理。处理后将返回响应。
  • 再次通过过滤器
  • 返回用户

而在spring中的请求与响应过程如下

Spring框架中，处理HTTP请求的过程是非常核心的功能，它是基于servlet的。

  • 客户端发出请求

  • 请求到达DispatcherServlet

    • 请求首先由Spring的核心组件DispatcherServlet接收。DispatcherServlet是一个前端控制器（Front Controller），负责将请求分发给适当的处理器。
    • 配置：DispatcherServlet通常在web.xml中配置。

    在Spring Boot中，DispatcherServlet由框架自动配置，无需手动设置。

  • Handler Mapping映射请求

    DispatcherServlet接收到请求后，通过一个或多个HandlerMapping组件来确定哪个控制器（Controller）应处理该请求。HandlerMapping根据请求URL、HTTP方法以及可能的其他参数来查找合适的处理器。

    • 实现：@RequestMapping、@GetMapping等注解用于映射请求到控制器方法。

  • HandlerAdapter调用处理器

    • 一旦找到合适的处理器（通常是一个控制器方法），DispatcherServlet使用HandlerAdapter来调用处理器。HandlerAdapter负责执行处理器的业务逻辑
    • 处理器：处理器通常是一个带有@Controller或@RestController注解的类，方法上可能有@RequestMapping、@GetMapping、@PostMapping等注解。控制器方法通常会调用业务逻辑层（Service层）进行实际的业务处理，业务逻辑层可能进一步调用数据访问层（DAO层）与数据库交互。这种分层架构使得代码结构清晰、职责分离，便于维护和扩展。

  • 响应发送回前端

    • 处理完成，后端通常通过RESTful API（一般基于ResponseEntity）向前端提供数据

    此外

    • 拦截器：Spring MVC允许在处理请求的各个阶段使用拦截器（Handler Interceptor）。拦截器可以在请求到达控制器之前、视图渲染之后执行额外的逻辑，如权限验证、日志记录等。

    • 异常处理：Spring MVC提供了全局异常处理机制，可以通过@ControllerAdvice和@ExceptionHandler注解来处理控制器方法中抛出的异常，并生成适当的响应。

总结

springMVC基于servlet，核心是DispatcherServlet，Handler Mapping，HandlerAdapter三个类。Handler Mapping负责路径映射，找到HandlerAdapter，然后HandlerAdapter调用处理器。
